# Backend

This package contains the FastAPI service behind Docgate. It manages user state, prepaid codes, Stripe fulfillment, and the internal auth endpoints that Nginx uses to protect static docs.

The backend is intentionally narrow: it does not try to serve the full website. Its main job is to answer identity and entitlement questions quickly and reliably.

## Responsibilities

- maintain local user and prepaid-code data
- sync doc-reading permissions into SuperTokens roles
- provide authenticated user and admin APIs
- handle Stripe checkout lifecycle
- expose fast internal auth endpoints for Nginx
- emit structured logs suitable for syslog collection

## Tech Stack

- FastAPI
- SQLAlchemy asyncio
- PostgreSQL via `asyncpg`
- SuperTokens Python SDK
- Stripe Python SDK
- Gunicorn + Uvicorn workers

## Core APIs

### User APIs

- `GET /api/user/get-supertokens-info`
- `GET /api/user/get`
- `POST /api/user/purchase-by-code`

### Admin APIs

- `POST /api/admin/gen-prepaid-code`
- `POST /api/admin/create-password-reset-link`
- `POST /api/admin/manually-verify-email`

### Internal Auth APIs

- `GET /api/internal-auth/check`
- `GET /api/internal-auth/refresh-session-or-signin`

### Stripe APIs

- `POST /api/stripe/create-checkout-session`
- `GET /api/stripe/session-status`
- `POST /api/stripe/fulfill-checkout-webhook`
- `POST /api/stripe/after-pay`

## Data Model

The backend keeps a small local domain model:

- `User`: local mirror of app-level user state, tier, lifetime, and payment log
- `PrepaidCode`: redeemable invite / purchase code with expiration and binding state

The database is the source of truth for payment state, while SuperTokens session claims are used as the fast authorization path during requests.

## Authentication and Authorization Design

### Normal API Auth

Regular API endpoints use SuperTokens session verification.

### Protected Docs Auth

For Nginx-protected docs routes, the backend uses a faster internal path:

```text
Nginx auth_request
  -> /api/internal-auth/check
  -> verify access token locally via JWKS
  -> check email verification claim
  -> check doc-reading role claim
  -> return 200 / 401 / 403
```

This avoids a core round-trip for every protected docs request and is one of the most important performance choices in the project.

## Business Flows

### Prepaid Code Redemption

- validate code existence and expiry
- lock the code row with `FOR UPDATE`
- create the user if needed
- mark the code as used
- update local tier to paid
- append payment log
- sync doc-reading role into SuperTokens

### Stripe Purchase

- create checkout session from the authenticated user
- receive webhook on payment completion
- mark local user as paid
- complete post-payment role sync in `after-pay`
- send purchase confirmation email

## Environment

Backend runtime config is loaded from env files generated by `confgen`:

- shared client-facing values such as base paths and domains
- server-only secrets such as DB, SMTP, SuperTokens, and Stripe credentials

The app requires `ENV` to be set to one of:

- `dev`
- `staging`
- `prod`

## Operational Notes

- The default production process model is one Gunicorn worker, which is intentional for low-resource VPS deployment.
- Logging is built for syslog aggregation.
- The JWT verification path maintains an in-memory JWKS cache to reduce repeated key fetches.
- Session claims are updated after successful purchase or code redemption so that authorization can stay fast at request time.
