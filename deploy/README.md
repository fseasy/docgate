# Deploy

This directory contains the shell-based deployment flow for Docgate. It is designed for a small Linux VPS with Nginx and systemd, and does not require Docker, avoiding container overhead if the machine is very small.

For the intended Linux setup, deployment is basically centered around two scripts:

```bash
cd deploy
bash env_init.sh
bash project_init_and_update.sh --mode serving
```

`env_init.sh` is the one-time machine bootstrap. `project_init_and_update.sh` is the repeatable deploy/update entry.

The deployment model is intentionally simple:

- Nginx serves as the public entrypoint
- the frontend is built as static assets
- the backend runs under Gunicorn + systemd
- config files are generated by `confgen`

## Target Scenario

This setup is a good fit for:

- a low-resource VPS
- one main application host
- operators who prefer shell scripts and systemd over containers

It is not trying to be a general-purpose cluster deployment framework.

## Deployment Architecture

```text
Internet
  |
  v
Nginx
  |-- serves public/static content
  |-- serves frontend build output
  |-- proxies /api/... to Gunicorn/FastAPI on 127.0.0.1:3001
  `-- protects /docs/... with auth_request

systemd
  `-- manages docgate-fastapi.service

Gunicorn
  `-- runs FastAPI with 1 worker by default
```

## Prerequisites

Before using these scripts, make sure the machine already has or can access:

- Ubuntu/Debian-style system utilities
- Nginx
- systemd
- a reachable SuperTokens Core instance (You can get one free from the official website, or you can deploy 1 by yourself.)
- a reachable PostgreSQL-compatible database (I use supabase free tier.)
- SMTP credentials (I use wechat email.)
- Stripe credentials if payment is enabled

## Deployment Flow

### 1. Bootstrap the Machine

Run:

```bash
cd deploy
bash env_init.sh
```

What it does:

- installs `uv` if missing
- installs `pnpm` if missing
- installs `git` if missing
- writes deploy env values to `deploy/.env`
- generates the `docgate-fastapi.service` file

You can override machine-specific values through `deploy/.machine.env`.

## 2. Prepare Unified Config

The deployment scripts expect environment-specific config to exist for `confgen`.

For open-source usage, the simplest approach is:

```bash
cd confgen
mkdir -p src/docgate_confgen/unified_conf/prod
cp src/docgate_confgen/unified_conf/example/conf.py src/docgate_confgen/unified_conf/prod/conf.py
```

Then edit `prod/conf.py` for your own domains, paths, and credentials.

## 3. Build and Deploy the Project

Run:

```bash
cd deploy
bash project_init_and_update.sh --mode serving
```

What it does:

- syncs the project repo to the `release` branch in serving mode
- prepares `confgen`
- generates backend env files, frontend env files, and Nginx config
- installs frontend dependencies and builds the SPA
- installs backend dependencies
- installs or updates the systemd service
- links generated Nginx config into the system config directory
- runs `nginx -t`
- restarts Nginx

## Modes

`project_init_and_update.sh` supports two modes:

- `serving`: production-style mode, intended for the VPS
- `deving`: skip forced git sync so local changes are preserved

## Runtime Details

### Gunicorn

The default Gunicorn configuration:

- binds to `127.0.0.1:3001`
- uses `uvicorn.workers.UvicornWorker`
- runs with `workers = 1`
- disables the control socket to avoid a known hang / deadlock issue

This is a deliberate choice for low-memory machines.

### systemd

The generated service:

- starts the FastAPI backend from the project virtualenv (created by uv)
- exports `ENV`
- exports syslog-related variables
- restarts automatically on failure, with db connection watchdog.

### Nginx

The generated Nginx config is expected to:

- serve the public site and SPA
- proxy `/api/` to the backend
- protect docs routes with `auth_request`
- redirect users to sign-in or purchase pages when needed

## Recommended Manual Checks

After deployment, verify:

- protected docs routes redirect correctly for anonymous users
- paid users can access protected docs
